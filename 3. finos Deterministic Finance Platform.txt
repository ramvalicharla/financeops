6. Detection pipeline
This should be deterministic.

Step 1 — File intake

Step 2 — Sheet profiling

header candidates

blank row density

merged cells presence

likely data block ranges

formula density

text vs numeric ratio

Step 3 — Structure detection
Identify:

data start row

section breaks

subtotal rows

period columns

dimension columns

notes columns

Step 4 — Signature generation
Create deterministic signatures:

header signature

row label signature

section signature

column order signature

Step 5 — Template match
Compare with existing effective template versions.

Outcome:

exact match

match with tolerable drift

no match → candidate new template version

Step 6 — Mapping resolution
Map:

source columns → canonical dimensions / roles

row labels → canonical metrics

subtotal patterns → section hierarchy

Step 7 — Validation
Check:

required metrics present

required periods present

numeric consistency

duplicate dimensional grain

subtotal sanity

sign consistency

Step 8 — Snapshot creation
Create immutable snapshot and normalized lines.

7. Drift handling policy
This must be strict.

Minor drift examples
header wording changed slightly

one optional note column added

column moved but identifiable

cosmetic blank row inserted

These may be auto-accepted if confidence is high.

Major drift examples
metric rows renamed materially

new section inserted

period axis changed

pivot structure changed

dimensions changed

totals logic changed

These must:

create drift event

fail closed or hold in review

not silently load into existing template

That is the audit-safe approach.

Phase 1F.3 — Payroll / GL Normalization Layer
Audit-Grade Design
1. Purpose
Phase 1F.3 creates the platform’s canonical normalization layer for two critical operational sources:

Payroll data

General Ledger dumps / transaction extracts

Its purpose is to take messy source files or source API payloads and convert them into a stable, append-only, tenant-isolated, drillable canonical representation that downstream modules can trust.

This layer exists so later modules do not have to repeatedly decode raw payroll and GL structures.

2. Why this phase matters
Without normalization:

reconciliation becomes brittle

MIS mapping becomes inconsistent

payroll-to-employee-cost checks become unreliable

schedule-to-GL controls become hard to automate

finance analytics become client-format dependent

With normalization:

one canonical structure supports many downstream uses

line-level drillability is preserved

source lineage remains intact

mapping logic becomes reusable

reconciliation quality improves materially

So this phase is not a cosmetic ETL layer.
It is the financial data contract layer.

3. Architectural position
Phase 1F.3 sits between:

Inputs
payroll exports

GL dumps

ERP transaction extracts

manual upload files

future accounting APIs

Outputs
canonical payroll lines

canonical GL lines

normalized dimension mappings

source lineage references

Consumers
Reconciliation Bridge (1F.2)

ratio / variance engine

board pack feeder

future close-control workflows

future CFO assistant layer

It must not directly mutate:

accounting engine tables

The design must obey the platform doctrine.
