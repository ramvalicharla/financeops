## 2. Authentication Flow

### Token Architecture
```
Access Token:   JWT, 15 minute expiry, contains: user_id, tenant_id, org_id, role, jti
Refresh Token:  JWT, 7 day expiry, stored in httpOnly cookie
CSRF Token:     For state-changing operations, stored in memory (not localStorage)
```

### Login Flow
```
1. POST /auth/login { email, password, mfa_code? }
   Response: { access_token, token_type: "bearer", expires_in: 900 }
   Cookie set: refresh_token (httpOnly, Secure, SameSite=Strict)

2. Frontend stores access_token in memory (Zustand store)
   NEVER in localStorage or sessionStorage

3. Every API request includes:
   Authorization: Bearer {access_token}
   X-Correlation-ID: {uuid-generated-per-request}
   X-CSRF-Token: {csrf-token} (for mutations)

4. Access token expires → silent refresh:
   POST /auth/refresh (refresh_token sent via cookie automatically)
   Response: new access_token
   Frontend updates Zustand store

5. Refresh token expires → redirect to login
```

### Token Refresh (Frontend Implementation)
```typescript
// axios interceptor pattern
axiosInstance.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401 && !error.config._retry) {
      error.config._retry = true;
      try {
        const { data } = await axios.post('/auth/refresh');
        useAuthStore.getState().setAccessToken(data.access_token);
        error.config.headers.Authorization = `Bearer ${data.access_token}`;
        return axiosInstance(error.config);
      } catch {
        useAuthStore.getState().logout();
        window.location.href = '/login';
      }
    }
    return Promise.reject(error);
  }
);
```

## 3. API Design Conventions

### URL Structure
```
/api/v1/{resource}                    List / Create
/api/v1/{resource}/{id}               Read / Update (new version) / Delete (soft)
/api/v1/{resource}/{id}/{sub-resource} Nested resources
/api/v1/{module}/{action}             Actions (non-CRUD)
```

### Standard Request Headers
```
Authorization:    Bearer {access_token}         [Required on all authenticated routes]
X-Correlation-ID: {uuid}                         [Required — generated by frontend per request]
X-Tenant-ID:      {tenant_uuid}                  [Verified against JWT — extra safety layer]
Content-Type:     application/json               [For JSON body]
X-CSRF-Token:     {csrf_token}                   [Required for POST/PUT/DELETE]
```

### Standard Response Envelope
```json
{
  "success": true,
  "data": { },
  "meta": {
    "correlation_id": "uuid",
    "timestamp": "2025-03-31T22:14:03Z",
    "version": "1.0",
    "pagination": {
      "page": 1,
      "page_size": 50,
      "total": 234,
      "total_pages": 5
    }
  },
  "errors": null
}
```

### Error Response
```json
{
  "success": false,
  "data": null,
  "meta": {
    "correlation_id": "uuid",
    "timestamp": "2025-03-31T22:14:03Z"
  },
  "errors": [
    {
      "code": "VALIDATION_ERROR",
      "field": "amount",
      "message": "Amount must be a positive number",
      "detail": null
    }
  ]
}
```

### HTTP Status Codes Used
```
200 OK              Successful GET, successful action
201 Created         Successful POST (new resource created)
202 Accepted        Request accepted, processing async (returns job_id)
400 Bad Request     Validation error (details in errors array)
401 Unauthorized    JWT missing or invalid
403 Forbidden       JWT valid but insufficient permissions
404 Not Found       Resource not found (or tenant isolation — same response)
409 Conflict        Version conflict (optimistic locking)
422 Unprocessable   Business logic error (e.g. "Cannot approve — reviewer step not complete")
429 Too Many Requests Rate limit exceeded
500 Internal Error  Unexpected server error (logged to Sentry)
503 Unavailable     Service temporarily unavailable (maintenance or overload)
```

## 5. Real-Time Communication

### Server-Sent Events (SSE) — AI Streaming
```typescript
// Frontend: consuming SSE stream
const streamAIQuery = async (query: string) => {
  const eventSource = new EventSource(
    `/api/v1/ai/query/stream?q=${encodeURIComponent(query)}`,
    { withCredentials: true }
  );

  eventSource.onmessage = (event) => {
    const chunk = JSON.parse(event.data);
    // chunk types: token | stage_update | confidence | done | error
    if (chunk.type === 'token') appendToResponse(chunk.content);
    if (chunk.type === 'stage_update') updateStageIndicator(chunk.stage, chunk.status);
    if (chunk.type === 'confidence') updateConfidenceBar(chunk.score);
    if (chunk.type === 'done') eventSource.close();
    if (chunk.type === 'error') handleError(chunk.error);
  };
};
```

### Long-Running Jobs — Polling Pattern
```typescript
// For async operations (report generation, ERP sync, consolidation run)
const runConsolidation = async (period: string) => {
  // 1. Trigger job → get job_id
  const { data } = await api.post('/api/v1/consolidation/run', { period });
  const jobId = data.data.job_id;

  // 2. Poll for completion
  const result = await pollJob(jobId);
  return result;
};

const pollJob = async (jobId: string, maxAttempts = 60) => {
  for (let i = 0; i < maxAttempts; i++) {
    const { data } = await api.get(`/api/v1/jobs/${jobId}`);
    if (data.data.status === 'complete') return data.data.result;
    if (data.data.status === 'failed') throw new Error(data.data.error);
    await new Promise(r => setTimeout(r, 2000)); // poll every 2s
  }
  throw new Error('Job timeout');
};
```

## 8. Error Handling Contract

### Frontend Error Handling
```typescript
// Global API error handler (in TanStack Query)
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: (failureCount, error: any) => {
        // Don't retry on 4xx errors
        if (error.response?.status < 500) return false;
        return failureCount < 3;
      },
    },
    mutations: {
      onError: (error: any) => {
        const correlationId = error.response?.headers['x-correlation-id'];
        const errors = error.response?.data?.errors;

        if (errors?.[0]?.code === 'VALIDATION_ERROR') {
          // Show field-level errors on form
          showFieldErrors(errors);
        } else if (error.response?.status === 401) {
          // Token refresh handled by interceptor — if we get here, redirect to login
          redirectToLogin();
        } else if (error.response?.status === 403) {
          showToast('You do not have permission to perform this action');
        } else if (error.response?.status === 429) {
          showToast('Too many requests. Please wait a moment.');
        } else {
          // Show generic error with correlation ID for support
          showToast(`An error occurred. Reference: ${correlationId}`);
          captureException(error, { extra: { correlationId } }); // Sentry
```

## 16. WebSocket Reconnection Strategy

```
CLIENT-SIDE RECONNECTION (frontend/hooks/useWebSocket.ts):

const reconnectWithBackoff = () => {
    const delays = [1000, 2000, 4000, 8000, 16000, 30000, 60000];
    let attempt = 0;

    const connect = () => {
        const ws = new WebSocket(WS_URL);
        
        ws.onclose = (event) => {
            if (event.code === 1000) return;  // clean close, don't reconnect
            
            const delay = delays[Math.min(attempt, delays.length - 1)];
            attempt++;
            
            // Show user: "Connection lost. Reconnecting in Xs..."
            showReconnectBanner(delay / 1000);
            
            setTimeout(connect, delay);
        };
        
        ws.onopen = () => {
            attempt = 0;  // reset backoff on successful connect
            hideReconnectBanner();
            
            // STATE RECOVERY: request missed events since last connection
            ws.send(JSON.stringify({
                type: 'sync_request',
                last_event_id: getLastReceivedEventId(),
                tenant_id: getTenantId()
            }));
        };
        
        return ws;
    };
    
    return connect();
};

SERVER-SIDE: on sync_request:
  Send all events since last_event_id for this tenant.
  Events stored in Redis stream with TTL 1 hour.
  If last_event_id too old: send current state snapshot.

OFFLINE QUEUE:
  Max size: 50 items.
  If offline and user takes action:
    Queue action locally (IndexedDB, encrypted).
    Show: "Action queued — will sync when reconnected"
  On reconnect: flush queue in order.
  On queue overflow (>50): warn user "Offline limit reached.
    Some actions may not have been saved."
```

## 18. Concurrent Token Refresh (Race Condition Prevention)

```
PROBLEM: 5 API calls fire simultaneously. JWT is expired.
  All 5 get 401. All 5 try to refresh. Only 1 succeeds.
  Result: 4 failed refreshes, confused state.

SOLUTION: Token refresh mutex (frontend/utils/auth.ts):

let refreshPromise: Promise<string> | null = null;

async function getValidToken(): Promise<string> {
    const token = getStoredToken();
    
    if (!isExpired(token)) {
        return token;  // fast path — token valid
    }
    
    // If refresh already in progress: wait for it (don't start another)
    if (refreshPromise) {
        return refreshPromise;
    }
    
    // Start refresh — store promise so concurrent calls can wait on it
    refreshPromise = refreshToken()
        .then(newToken => {
            storeToken(newToken);
            refreshPromise = null;  // clear mutex
            return newToken;
        })
        .catch(err => {
            refreshPromise = null;  // clear mutex on error too
            redirectToLogin();
            throw err;
        });
    
    return refreshPromise;
}

// All API calls use this:
const token = await getValidToken();
fetch(url, { headers: { Authorization: `Bearer ${token}` } });

RESULT:
  5 concurrent calls → 1 refresh → all 5 get new token → all 5 succeed.
  Zero duplicate refresh calls.
  Zero confused state.
```
